📋 ১. প্রেক্ষাপট (Context)

২০২৬ সালের প্রযুক্তি শাসন রাজ্যে আমরা দাঁড়িয়ে — যেখানে AI মানুষের পকেটে, আর মানুষ কোন কিছু করার আগে AI-র পরামর্শ নেয়। এই বাস্তবতায় ZombieMCP নামে একটি Multi-Provider MCP বানানো হচ্ছে যেখানে:

  • ৬ টি AI Provider (GroqBridge, OpenCode, Google, Ollama, Groq, Local)
  • ৭ টি MCP Tool (ping_agent, run_agent, list_providers, ...)
  • Live Dashboard + Auth Middleware + AES-256-GCM Encryption
  • VS Code, IntelliJ IDEA, Hermes Agent MCP Integration

🔍 ২. প্রথম বিশ্লেষণ — আমার দেওয়া (যা ভুল ছিল)

আমি ৩ টি সমস্যা চিহ্নিত করেছিলাম:

  1. mcp_clients টেবিল অসম্পূর্ণ — session_id কে PRIMARY KEY বানানো
  2. Verification flow নেই — ক্লায়েন্ট approve/reject করার উপায় নেই
  3. Hermes Agent MCP disconnected — env filter + restart না করায়

আর আমি প্রস্তাব দিয়েছিলাম 3-way solution:

  • ✅ GUI clients (VS Code, IntelliJ) → Browser link
  • ✅ CLI clients (Hermes) → API key–based verification (GitHub CLI-র মতো)
  • ✅ Dashboard → Manual approve button

⚠ আমি ভুল ছিলাম। আপনি ধরিয়ে দিলেন — আমি শুধু "এক্সপেরিয়েন্স" চিন্তা করেছি, "ফান্ডামেন্টাল" চিন্তা করিনি।

🎯 ৩. আপনার কাউন্টার — ফান্ডামেন্টাল ট্রুথ

— ভাইয়া (Sahon Srabon)

🔑 মূল পয়েন্টগুলো

📍 পয়েন্ট ১

VS Code, IntelliJ, Hermes — সবগুলো ইলেকট্রন অ্যাপ। দিন শেষে এগুলো ওয়েব ব্রাউজার। "লোকাল" বলে কিছু নেই।

📍 পয়েন্ট ২

আমরা একটা চাবি (key) তৈরির সিস্টেম বানাচ্ছি। চাবি সঠিক হলে দরজা খুলবে। এটাই মূল কাজ — authentication function।

📍 পয়েন্ট ৩

সেশন রিকোয়েস্ট এলে ব্রাউজারে পোলিং করবে, ব্রাউজার থেকে মাল-মসলা নিবে, callback function return করবে। তখনই ক্লায়েন্ট জানে আপনি authentic user।

📍 পয়েন্ট ৪

আমি শুধু আমার দিকটা রক্ষা করেছি (server-side auth)। কিন্তু ক্লায়েন্টের দিক থেকে — ওরা এখনো জানে না আমি কে। ওদের বিশ্বাস করাইনি।

❌ আমি যেখানে ভুল করেছি

আমি ৩ টি ভিন্ন উপায় (browser, API key, button) বলেছি — কিন্তু ভুল ফান্ডামেন্টাল ধরেছিলাম। আপনার সঠিক পথ:

  1. সবার জন্য browser-based verification — কারণ সব EDTOR-ই ওয়েব (Electron)
  2. Auto-detect → auto-open browser — ইউজারকে VS Code ওপেন করতেই হবে না
  3. Server নিজেই verify করবে — ইউজার ভুলে গেলেও সার্ভার ভুলবে না
  4. LocalStorage 24h — ব্রাউজারে session token সংরক্ষণ

🔄 Verification Flow (Step by Step)

  1. MCP Client (VS Code/Hermes/IntelliJ) → প্রথমবার ZombieMCP-তে connect করে
  2. Server → client_sessions টেবিলে নতুন এন্ট্রি (status: pending)
  3. Server → ৬-ডিজিটের verification code জেনারেট করে
  4. Client → Browser-এ /verify/{code} URL automatically ওপেন করে
  5. Browser → ইউজারকে দেখায়: "VS Code থেকে সংযোগের অনুরোধ"
  6. ইউজার → [Approve] ক্লিক করে
  7. Browser → localStorage-এ session token সংরক্ষণ করে (২৪ ঘন্টা)
  8. Server → status verified করে + callback পাঠায় client-এ
  9. Client → এখন tool call করতে পারে + heartbeat শুরু করে
  10. পরবর্তী connect-এ → localStorage থেকে token পড়ে + server verify করে

✅ সঠিক উত্তর

আপনি ঠিক বলেছেন — Hermes নিজেও Electron-based। Hermes Desktop অ্যাপ = একটা ব্রাউজার। Hermes CLI-তেও hermes auth কমান্ড আছে যেটা ব্রাউজার খুলে OAuth করে। তাই সবার জন্যই Browser verification করা সম্ভব — API key–based আলাদা পথের দরকার নেই।

❌ আমার আরেকটি ভুল

আমি বলেছিলাম: "Dashboard-এ manual approve button — সবচেয়ে সহজ সমাধান"

✅ সঠিক উত্তর

বাটন হেল্পার হিসেবে থাকতে পারে — emergency approve-এর জন্য। কিন্তু মূল মেকানিজম automatic browser verification হওয়া উচিত। বাটন প্রকৃত সমাধান না — এটা কাজ করে শুধুমাত্র ইউজার active থাকলে। automatic সিস্টেম ইউজার ভুলে গেলেও কাজ করবে।

📊 ৬. ডাটাবেজ স্কিমা — প্রস্তাবিত পরিবর্তন

-- 🆕 NEW: client_sessions (replace mcp_clients)
CREATE TABLE client_sessions (
    id                TEXT PRIMARY KEY,        -- UUID (independent of transport)
    client_id         TEXT NOT NULL,           -- unique client identifier (auto-generated)
    client_name       TEXT NOT NULL,           -- "Visual Studio Code", "Hermes", etc.
    client_version    TEXT,
    client_type       TEXT,                    -- 'vscode' | 'intellij' | 'hermes' | 'opencode' | 'other'
    working_directory TEXT,
    status            TEXT NOT NULL DEFAULT 'pending',  -- pending | verified | active | disconnected
    verification_code TEXT,                    -- 6-digit one-time code
    verified_at       INTEGER,
    ip_address        TEXT,
    user_agent        TEXT,
    connected_at      INTEGER NOT NULL DEFAULT (unixepoch()),
    last_seen         INTEGER NOT NULL DEFAULT (unixepoch()),
    expires_at        INTEGER,                 -- 24h from verification
    conversation_count INTEGER DEFAULT 0,
    metadata_json     TEXT DEFAULT '{}'
);

-- 🆕 NEW: conversations (group agent_runs under a session)
CREATE TABLE conversations (
    id                TEXT PRIMARY KEY,        -- UUID
    client_session_id TEXT NOT NULL,           -- FK → client_sessions.id
    title             TEXT,
    status            TEXT DEFAULT 'active',   -- active | archived
    started_at        INTEGER NOT NULL DEFAULT (unixepoch()),
    last_active       INTEGER NOT NULL DEFAULT (unixepoch()),
    prompt_count      INTEGER DEFAULT 0
);

-- 🔄 UPDATE: agent_runs (link to conversation instead of raw session)
ALTER TABLE agent_runs ADD COLUMN conversation_id TEXT REFERENCES conversations(id);

-- 📌 INDEXES
CREATE INDEX idx_client_sessions_status ON client_sessions(status);
CREATE INDEX idx_conversations_session ON conversations(client_session_id);
